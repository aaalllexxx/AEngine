## Кодекс поведения

### Наши обязательства

Мы стремимся создать открытое и дружелюбное сообщество. Мы обязуемся:

- Использовать приветливый и инклюзивный язык
- Уважать различные точки зрения и опыт
- Принимать конструктивную критику
- Фокусироваться на том, что лучше для сообщества
- Проявлять эмпатию к другим участникам

### Неприемлемое поведение

- Использование сексуализированного языка или изображений
- Троллинг, оскорбительные комментарии, личные атаки
- Публичные или приватные домогательства
- Публикация личной информации других без разрешения
- Другое поведение, которое можно считать неуместным

### Ответственность

Мейнтейнеры проекта имеют право удалять, редактировать или отклонять комментарии, коммиты, код, правки wiki, issues и другие вклады, которые не соответствуют этому Кодексу поведения.

## Как я могу помочь?

### Сообщения об ошибках

Нашли баг? Создайте issue на GitHub:

1. Проверьте, что issue еще не создан
2. Используйте понятный заголовок
3. Опишите шаги для воспроизведения
4. Укажите ожидаемое и фактическое поведение
5. Добавьте информацию о системе (ОС, версия Python)

**Шаблон issue для бага:**

```markdown
## Описание
Краткое описание проблемы.

## Шаги для воспроизведения
1. Перейти в '...'
2. Нажать на '...'
3. Увидеть ошибку

Что должно было произойти.

## Фактическое поведение
Что произошло на самом деле.

## Окружение
- ОС: Windows 10
- Python: 3.9.5
- AEngine: 2.2

## Дополнительная информация
Логи, скриншоты, и т.д.
```

## Процесс внесения изменений

### 1. Fork и клонирование

```bash
# Fork репозитория на GitHub
# Затем клонируйте свой fork
git clone https://github.com/YOUR_USERNAME/AEngine.git
cd AEngine

# Добавьте upstream remote
git remote add upstream https://github.com/ORIGINAL_OWNER/AEngine.git
```

### 2. Создание ветки

```bash
# Обновите main
git checkout main
git pull upstream main

# Создайте ветку для вашей работы
git checkout -b feature/my-awesome-feature
```

**Именование веток:**

- `feature/` — новая функциональность
- `fix/` — исправление бага
- `docs/` — изменения в документации
- `refactor/` — рефакторинг кода
- `test/` — добавление тестов

### 3. Настройка окружения

```bash
# Создайте виртуальное окружение
python -m venv venv

# Активируйте его
# Windows
venv\Scripts\activate
# Linux/macOS
source venv/bin/activate

# Установите зависимости
pip install -r AEngineApps/requirements.txt
pip install -r APM/requirements.txt

# Установите инструменты разработки
pip install pytest pytest-cov black flake8 mypy
```

### 4. Внесение изменений

- Пишите чистый, понятный код
- Следуйте стандартам кодирования (см. ниже)
- Добавляйте тесты для новой функциональности
- Обновляйте документацию
- Делайте небольшие, логичные коммиты

```bash
# Проверьте изменения
git status
git diff

# Добавьте файлы
git add path/to/file

# Сделайте коммит
git commit -m "feat: Add awesome feature"
```

### 5. Тестирование

```bash
# Запустите тесты
pytest

# Проверьте покрытие
pytest --cov=AEngineApps --cov=APM --cov=sec

# Проверьте стиль кода
black .
flake8 . --max-line-length=100

# Проверьте типы
mypy AEngineApps/ --ignore-missing-imports
```

### 6. Push и Pull Request

```bash
# Push в ваш fork
git push origin feature/my-awesome-feature
```

Затем на GitHub:

1. Перейдите в ваш fork
2. Нажмите "New Pull Request"
3. Выберите вашу ветку
4. Заполните описание PR (см. шаблон ниже)
5. Нажмите "Create Pull Request"

**Шаблон Pull Request:**

```markdown
## Описание
Краткое описание изменений.

## Тип изменения
- [ ] Исправление бага (non-breaking change)
- [ ] Новая функция (non-breaking change)
- [ ] Breaking change (изменение, ломающее обратную совместимость)
- [ ] Документация

## Связанные issues
Fixes #123
Related to #456

## Как протестировано?
Опишите тесты, которые вы провели.

## Checklist
- [ ] Код следует стандартам проекта
- [ ] Проведен self-review кода
- [ ] Добавлены комментарии для сложных участков
- [ ] Обновлена документация
- [ ] Изменения не генерируют новых warnings
- [ ] Добавлены тесты
- [ ] Все тесты проходят
- [ ] Обновлен CHANGELOG.md
```

### 7. Code Review

После создания PR:

1. Автоматические проверки (CI/CD) должны пройти
2. Мейнтейнеры проведут review
3. Отвечайте на комментарии
4. Вносите запрошенные изменения
5. Push обновлений в ту же ветку

```bash
# Внесите изменения
git add .
git commit -m "fix: Address review comments"
git push origin feature/my-awesome-feature
```

### 8. Merge

После одобрения:

- Мейнтейнер сделает merge вашего PR
- Ваша ветка будет удалена
- Изменения попадут в main

```bash
# Обновите ваш fork
git checkout main
git pull upstream main
git push origin main

# Удалите локальную ветку
git branch -d feature/my-awesome-feature
```

## Стандарты кода

### Python Style

Следуем **PEP 8** с некоторыми модификациями:

- Максимальная длина строки: **100 символов**
- Используем **4 пробела** для отступов
- Используем **двойные кавычки** для строк

### Форматирование

Используем **Black** для автоматического форматирования:

```bash
black AEngineApps/ APM/ sec/
```

### Линтинг

Используем **Flake8**:

```bash
flake8 . --max-line-length=100 --exclude=venv,__pycache__
```

### Type Hints

Используем type hints везде, где возможно:

```python
def process_user(user_id: int, name: str) -> dict:
    """Обрабатывает пользователя."""
    return {"id": user_id, "name": name}
```

### Docstrings

Используем **Google Style** docstrings:

```python
def complex_function(param1: str, param2: int) -> bool:
    """Краткое описание.
    
    Подробное описание функции.
    
    Args:
        param1: Описание параметра 1
        param2: Описание параметра 2
    
    Returns:
        Описание возвращаемого значения
    
    Raises:
        ValueError: Когда param2 < 0
    
    Example:
        >>> complex_function("test", 5)
        True
    """
    pass
```

### Именование

```python
# Классы: PascalCase
class UserManager:
    pass

# Функции и методы: snake_case
def get_user_data():
    pass

# Константы: UPPER_SNAKE_CASE
MAX_RETRIES = 3

# Приватные: _leading_underscore
def _internal_helper():
    pass
```

### Коммиты

Используем **Conventional Commits**:

```
<type>(<scope>): <subject>

<body>

<footer>
```

**Types:**
- `feat`: Новая функция
- `fix`: Исправление бага
- `docs`: Документация
- `style`: Форматирование (не влияет на код)
- `refactor`: Рефакторинг
- `test`: Добавление тестов
- `chore`: Обслуживание (обновление зависимостей и т.д.)
- `perf`: Улучшение производительности

**Примеры:**

```bash
git commit -m "feat(app): Add async support for screens"
git commit -m "fix(security): Fix SQL injection vulnerability"
git commit -m "docs(readme): Update installation instructions"
git commit -m "test(api): Add tests for API class"
```

### Тесты

Каждая новая функция должна иметь тесты:

```python
import pytest
from AEngineApps.app import App

class TestNewFeature:
    """Тесты для новой функции."""
    
    @pytest.fixture
    def app(self):
        """Создает тестовое приложение."""
        return App("TestApp")
    
    def test_feature_works(self, app):
        """Тестирует, что функция работает."""
        result = app.new_feature()
        assert result == expected_value
    
    def test_feature_handles_errors(self, app):
        """Тестирует обработку ошибок."""
        with pytest.raises(ValueError):
            app.new_feature(invalid_input)
```

## Процесс review

### Что проверяют reviewers

1. **Функциональность**
   - Код делает то, что заявлено?
   - Нет ли багов?
   - Обработаны ли edge cases?

2. **Качество кода**
   - Читаемость и понятность
   - Соответствие стандартам
   - Нет ли дублирования?

3. **Тесты**
   - Есть ли тесты?
   - Покрывают ли они функциональность?
   - Проходят ли все тесты?

4. **Документация**
   - Обновлена ли документация?
   - Есть ли docstrings?
   - Понятны ли примеры?

5. **Производительность**
   - Нет ли узких мест?
   - Оптимален ли алгоритм?

6. **Безопасность**
   - Нет ли уязвимостей?
   - Валидируются ли входные данные?

### Как отвечать на комментарии

- Будьте вежливы и профессиональны
- Объясняйте свои решения
- Если не согласны, аргументируйте
- Благодарите за feedback
- Вносите изменения быстро

**Хорошие ответы:**

```markdown
Спасибо за замечание! Исправил в последнем коммите.

Хороший вопрос. Я выбрал этот подход, потому что...

Согласен, это можно улучшить. Сделаю рефакторинг.
```

### Если PR отклонен

Не расстраивайтесь! Это нормальная часть процесса:

- Прочитайте причины отклонения
- Задайте вопросы, если что-то непонятно
- Внесите изменения и создайте новый PR
- Или обсудите альтернативные подходы
