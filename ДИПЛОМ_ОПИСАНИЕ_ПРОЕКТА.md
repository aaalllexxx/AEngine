# AEngine: подробное описание структуры и принципов работы проекта

## 1. Общее назначение проекта

`AEngine` представляет собой экосистему для разработки приложений на Python, в которой объединены:

- фреймворк `AEngineApps` для создания веб- и desktop-приложений;
- менеджер проектов и модулей `APM` (`AEngine Package Manager`);
- модуль информационной безопасности `sec`;
- набор документации, шаблонов и тестов для сопровождения и развития системы.

По сути, `AEngine` задуман как единая среда, которая позволяет:

1. создавать новые приложения по шаблону;
2. запускать их в синхронном или асинхронном режиме;
3. расширять приложение дополнительными модулями;
4. подключать механизмы безопасности;
5. управлять проектами через отдельный CLI-инструмент;
6. синхронизировать рабочую копию между WSL и Windows-хостом.

В рамках дипломной работы проект можно рассматривать как **прототип модульной платформы разработки защищенных приложений**, где разработчик получает не только базовый фреймворк, но и средства администрирования, автоматизации, тестирования и расширения функциональности.

## 2. Общая структура репозитория

В актуальном состоянии проект содержит несколько ключевых подсистем:

```text
AEngine/
├── AEngineApps/         # ядро фреймворка приложений
├── APM/                 # менеджер проектов и модулей
├── sec/                 # модуль безопасности
├── tests/               # автоматические тесты
├── README.md            # основное описание проекта
├── QUICK_START.md       # быстрый старт
├── API_REFERENCE.md     # описание API
├── ARCHITECTURE.md      # архитектурное описание
├── DEVELOPER_GUIDE.md   # руководство разработчика
```

3. **уровень защиты и контроля** — `sec`.

Дополнительно присутствует:

- документационный уровень;
- тестовый уровень;
- эксплуатационный уровень в виде Docker-конфигурации и пользовательского `wsl-sync`.

## 3. Ядро системы: AEngineApps

### 3.1. Роль AEngineApps

`AEngineApps` — это основной прикладной фреймворк. Он реализует логику построения приложения без использования привычной декораторной модели Flask, заменяя ее **объектно-ориентированным подходом**.

Идея фреймворка заключается в том, что разработчик описывает приложение через классы:

- `App` или `AsyncApp` — объект самого приложения;
- `Screen` или `AsyncScreen` — объект экранов и маршрутов;
- `Service` — объект логического сервисного модуля.

Это позволяет строить приложение как набор компонентов с понятной ответственностью.

### 3.2. Основные файлы AEngineApps

Ключевые файлы ядра:

- `app.py` — синхронное приложение на базе Flask;
- `async_app.py` — асинхронное приложение на базе Quart;
- `screen.py` — базовый синхронный экран;
- `async_screen.py` — базовый асинхронный экран;
- `service.py` — поддержка сервисов и модульной архитектуры;
- `api.py` — API-слой;
- `json_dict.py` — работа с JSON-конфигурацией;
- `global_storage.py` — общее хранилище данных.

### 3.3. Как работает App

Класс `App` отвечает за:

- создание экземпляра Flask;
- загрузку конфигурации приложения;
- регистрацию экранов;
- автоматическое обнаружение экранов;
- регистрацию сервисов;
- подключение middleware;
- настройку обработки ошибок;
- запуск в web-режиме или через `pywebview`.

Логика работы следующая:

1. создается объект `App`;
2. ему передается имя приложения;
3. из `config.json` читаются параметры работы;
4. при необходимости автоматически сканируются экраны;
5. каждый экран регистрируется как маршрут;
6. приложение запускает сервер.

Таким образом, `App` выступает **центральной точкой координации всего приложения**.

### 3.4. Как работает Screen

Класс `Screen` является базовым обработчиком пользовательского маршрута.

Каждый экран:

- наследуется от `Screen`;
- содержит атрибут `route`;
- содержит список HTTP-методов `methods`;
- реализует метод `run()`.

Например, если экран описывает главную страницу, он может вернуть HTML-шаблон. Если это API-экран, он может вернуть JSON-ответ.

Внутри `Screen` уже предусмотрены вспомогательные методы:

- `render()` — рендер HTML-шаблона;
- `json()` — формирование JSON-ответа;
- `redirect()` — перенаправление;
- `flash()` — flash-сообщения;
- `abort()` — аварийное завершение запроса;
- `save_file()` — сохранение загруженных файлов.

За счет этого разработчику не нужно каждый раз писать повторяющийся инфраструктурный код.

### 3.5. Как работает AsyncApp и AsyncScreen

Асинхронная ветка ядра реализована через:

- `async_app.py`;
- `async_screen.py`.

Она предназначена для сценариев, где:

- много сетевых операций;
- используется внешнее API;
- требуется масштабирование;
- важно неблокирующее выполнение.

`AsyncApp` по смыслу повторяет `App`, но:

- использует `Quart` вместо `Flask`;
- поддерживает `async def run()` в экранах;
- позволяет включать health endpoint;
- позволяет включать metrics endpoint;
- рассчитан на запуск через ASGI-серверы.

Это дает возможность использовать ту же архитектурную модель, но уже для high-load или I/O-intensive приложений.

### 3.6. Service и модульность

`service.py` реализует идею логических сервисов. Сервис — это отдельный модуль приложения, который может иметь:

- собственный префикс URL;
- собственные экраны;
- собственные middleware.

По сути, `Service` — это объектная надстройка над Flask `Blueprint`, но встроенная в общий стиль `AEngineApps`.

Такой подход позволяет разбивать приложение на подсистемы:

- авторизация;
- API;
- админ-панель;
- мониторинг;
- уведомления;
- логирование.

### 3.7. Конфигурация приложения

Приложение управляется через `config.json`.

Обычно в конфигурации задаются:

- `debug`;
- `view` (`web` или app/webview);
- `host`;
- `port`;
- `screen_path`;
- `routers`;
- пути к шаблонам и статике.

Класс `JsonDict` делает работу с конфигурацией удобной и централизованной.

## 4. Менеджер экосистемы: APM

### 4.1. Назначение APM

`APM` — это консольный менеджер, предназначенный для:

- создания новых проектов;
- запуска приложений;
- установки модулей;
- обновления фреймворка;
- управления зарегистрированными проектами.

С практической точки зрения `APM` делает из `AEngine` не просто набор Python-файлов, а **полноценную платформу с CLI-интерфейсом**.

### 4.2. Основной принцип работы APM

Файл `APM/apm.py` является точкой входа.

Он:

1. анализирует аргументы командной строки;
2. определяет нужную команду;
3. динамически загружает соответствующий модуль из `APM/modules`;
4. запускает его функцию `run(...)`.

Таким образом, `APM` реализован как модульная CLI-система.

### 4.3. Командная архитектура

Команды лежат в каталоге `APM/modules/`.

Ключевые команды:

- `create` — создание нового проекта;
- `run` — запуск проекта;
- `install` — установка модулей;
- `upgrade` — обновление `AEngineApps`;
- `modules` — просмотр установленных модулей;
- `config` — конфигурация проекта;
- `list`, `goto`, `select`, `unregister`, `delete`, `init` — управление проектами.

Каждый модуль команды:

- имеет текст справки;
- относится к своей функциональной группе;
- реализует отдельную функцию `run`.

Это делает CLI расширяемым.

### 4.4. Как работает apm create

Команда `apm create`:

1. создает директорию нового проекта;
2. копирует шаблон из `APM/examples/default_project`;
3. создает `.apm/run.json`;
4. подключает локальную копию `AEngineApps`;
5. регистрирует проект в глобальной конфигурации.

Важно, что в текущей реализации создание проекта больше не зависит от системного `git clone`, а может работать через локальный snapshot или GitHub archive. Это делает команду устойчивой в среде WSL и ограниченных окружениях.

### 4.5. Как работает apm run

Команда `apm run`:

1. определяет текущий проект;
2. читает `.apm/run.json`;
3. определяет главный файл проекта;
4. настраивает `PYTHONPATH`;
5. запускает приложение через текущий Python interpreter.

В актуальной версии команда:

- корректно использует активный интерпретатор;
- не выводит ложное сообщение об успехе при падении приложения;
- корректно обрабатывает остановку пользователем.

Это делает ее пригодной для реальной работы с проектами, созданными через `APM`.

### 4.6. Работа с модулями

Команда `apm install` устанавливает расширения в `.apm/installed/`.

Теперь она умеет:

- принимать shorthand вида `apm install sec`;
- загружать модуль через GitHub archive;
- использовать локальную sibling-копию модуля, если команда запущена из основной рабочей копии `AEngine`.

Это особенно важно для модуля `sec`, который теперь можно реально установить и использовать в сгенерированном проекте.

## 5. Модуль безопасности sec

### 5.1. Назначение sec

`sec` — это отдельная подсистема информационной безопасности, интегрируемая в приложения экосистемы `AEngine`.

Она предназначена для:

- обнаружения атак;
- анализа сетевого поведения;
- контроля целостности кода;
- защиты ОС и процесса исполнения;
- ведения логов;
- поддержки кластеризации;
- административного контроля.

С точки зрения дипломной работы именно `sec` делает проект не просто веб-фреймворком, а **платформой разработки защищенных приложений**.

### 5.2. Состав sec

Ключевые файлы модуля:

- `intrusions.py` — IDS/IPS и детекторы атак;
- `os_protect.py` — проверка состояния ОС и привилегий;
- `net_analyzer.py` — анализ сетевой активности;
- `sys_protect.py` — системная защита;
- `code_signer.py` — контроль целостности;
- `dashboard.py` — интерфейс администрирования;
- `cluster.py` и `auto_cluster.py` — кластеризация;
- `auth.py` — работа с администратором безопасности;
- `sign.py` и `unsign.py` — подпись и снятие подписи.

### 5.3. IDS/IPS

Подсистема IDS/IPS строится вокруг детекторов атак.

Поддерживаются детекторы для:

- SQL injection;
- XSS;
- RCE;
- LFI;
- rate limit сценариев.

Модуль может работать в двух режимах:

- **IDS** — только обнаружение и логирование;
- **IPS** — обнаружение и блокировка запроса.

Это позволяет использовать `sec` как встроенный слой прикладной защиты веб-приложения.

### 5.4. OS Protection и Network Analyzer

`os_protect.py` позволяет:

- определить привилегии процесса;
- получить информацию о загрузке системы;
- выявить потенциальные проблемы ресурсов.

`net_analyzer.py` позволяет:

- оценивать сетевую активность;
- искать аномальные IP;
- отслеживать условия, похожие на flood-атаки.

В ходе практической проверки эти модули были успешно импортированы и отработали базовый smoke-test.

### 5.5. sec как CLI-модуль APM

После установки `sec` в проект через `apm install sec` модуль становится subcommand-плагином.

Далее доступны команды вида:

- `apm sec init`
- `apm sec init --list`
- `apm sec sign`
- `apm sec unsign`
- `apm sec add_admin`

Это означает, что безопасность интегрирована не только как библиотека, но и как управляемый пользовательский инструмент.

## 8. Синхронизация между WSL и Windows

Для проекта настроена внешняя подсистема синхронизации `wsl-sync`.

Текущая рабочая схема:

- основная WSL-копия: `/home/alex/projects/AEngine`
- хостовая копия Windows: `/mnt/c/Users/Администратор/Desktop/AEngine`

Синхронизация работает так:

1. основным источником истины считается WSL-проект;
2. изменения в WSL автоматически передаются в хостовую папку;
3. служебные каталоги и артефакты не копируются;
4. демон работает в фоне как `systemd --user` сервис.

Исключаются:

- `.git`
- `.buildozer`
- `__pycache__`
- `.venv`
- `htmlcov`
- `coverage.xml`
- и другие служебные каталоги сборки/кэша.

Это важно для дипломной части, так как проект фактически поддерживает **двухсредную разработку**:

- Linux/WSL для основной инженерной работы;
- Windows-host для файлового присутствия, IDE-сценариев и внешней интеграции.

## 9. Архитектурные достоинства проекта

Проект `AEngine` интересен как объект дипломного исследования по нескольким причинам.

### 9.1. Модульность

Система разделена на независимые компоненты:

- фреймворк приложений;
- CLI-менеджер;
- security-модуль;
- тесты;
- документация.

Это позволяет развивать части проекта независимо.

Маршруты и обработчики представлены не функциями-декораторами, а полноценными классами. Это упрощает:

- сопровождение;

Через `APM` можно подключать внешние модули и плагины.

### 9.4. Поддержка async

Система поддерживает как синхронный, так и асинхронный режим работы, что делает ее применимой к широкому спектру сценариев.

### 9.5. Интеграция безопасности

Безопасность встроена в архитектуру не как внешний сервис, а как расширяемый модуль экосистемы.

## 10. Ограничения и практические замечания

Несмотря на рабочее состояние, проект имеет ряд ограничений, которые стоит честно указать в дипломной работе.

### 10.1. Не все подсистемы одинаково зрелые

Фреймворк, `APM` и базовые security-сценарии приведены в рабочее состояние, но часть документации описывает более широкий функционал, чем реально покрыто автоматическими тестами.

### 10.2. Android-сборка не является полностью верифицированной в текущем окружении

В репозитории присутствуют следы `buildozer` и документация по сборке APK, однако полный end-to-end запуск Android pipeline требует отдельного окружения.

### 10.3. Покрытие тестами пока ограничено

Основной автоматически проверенный блок — это async-часть `AEngineApps`. `APM` и `sec` подтверждены smoke-тестами и практическими сценариями, но не имеют такого же объема unit-тестов.

Эти ограничения не обнуляют проект, но являются важной частью инженерно честного описания.

С инженерной точки зрения `AEngine` можно охарактеризовать так:

> `AEngine` — это модульная программная платформа для разработки защищенных приложений на Python, включающая объектно-ориентированный прикладной фреймворк, менеджер проектов и модулей, а также интегрируемый модуль информационной безопасности.

Ключевая идея проекта заключается в том, что разработчик получает не отдельную библиотеку, а **целостную экосистему**, в которой объединены:

- разработка приложений;
- их конфигурирование;
- запуск и сопровождение;
- подключение модулей;
- базовые средства защиты;
- тестирование;
- эксплуатационная синхронизация рабочей среды.

Для дипломной работы проект можно представить как:

1. результат проектирования собственной прикладной платформы;
2. пример реализации модульной архитектуры на Python;
3. пример интеграции security-подсистемы в фреймворк приложений;
4. пример унификации разработки, запуска и сопровождения в единой системе.

---

## 12. Краткое резюме

`AEngine` состоит из трех центральных частей:

- `AEngineApps` — отвечает за логику приложений;
- `APM` — отвечает за управление проектами и модулями;
- `sec` — отвечает за безопасность и контроль.

Общий жизненный цикл работы с системой выглядит так:

1. через `APM` создается проект;
2. в проект подключается `AEngineApps`;
3. приложение описывается через `App/Screen` или `AsyncApp/AsyncScreen`;
4. при необходимости подключаются защитные подсистемы `sec`;
5. проект запускается, тестируется и синхронизируется с хостовой средой;
6. разработчик получает единую инженерную платформу, а не набор разрозненных файлов.

Именно это делает проект хорошей основой для дипломной работы по теме:

- разработки фреймворка;
- проектирования защищенной платформы;
- построения модульной архитектуры приложений;
- автоматизации жизненного цикла разработки.
